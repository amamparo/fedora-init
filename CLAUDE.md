# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Keep documentation in sync

When a change alters behavior, commands, role conventions, keybindings, or gotchas, update the documentation in the same change — this file, README.md, and any task-file header comments that describe the affected behavior. README.md is the user-facing doc (what the roles do and how to run them); CLAUDE.md is the contributor-facing doc (architecture, conventions, gotchas). A PR that changes what a role does but not the docs is incomplete.

## Commit and push at good stopping points

Whenever the work reaches a coherent, self-contained stopping point — a role added, a gotcha fixed, docs synced — commit and push without waiting to be asked. A "good stopping point" means the tree is consistent: docs match behavior (see above), `ansible-playbook site.yml --syntax-check` passes, and the change stands on its own. This repo does NOT use PRs or feature branches: commit straight to `main` with a descriptive message and `git push origin main`. Do NOT run the playbook to "verify" before committing — the safe checks in the section below are the gate.

## What this is

An Ansible playbook that configures a fresh Fedora Workstation (GNOME) install, targeting Fedora 44 / GNOME 50. The only entry point is:

```sh
./install.sh                # bootstrap the toolchain + run every role
./install.sh battery zsh    # run roles whose tag contains a substring
./install.sh --check        # dry run (any dash-arg passes through to ansible-playbook)
curl -fsSL https://raw.githubusercontent.com/amamparo/fedora-init/main/install.sh | bash   # no checkout
```

**Running the playbook mutates the host** (installs/removes dnf packages, masks systemd units, changes the login shell). Don't execute it to "test" a change unless the user asks — even `--check` prompts for sudo. The safe checks are:

```sh
ansible-playbook site.yml --syntax-check
ansible-lint --offline        # policy in .ansible-lint; passes the production profile
bash -n install.sh
python3 -m py_compile library/*.py
```

## Architecture

- `install.sh` — orchestrator and bootstrapper. If it can't find `site.yml` next to itself (piped from curl, or a stray lone copy), it downloads the GitHub tarball to a temp dir and re-runs from that copy. Otherwise it: reattaches stdin to `/dev/tty` (under curl|bash stdin is the exhausted pipe; the sudo prompt and the keyring pause both need the terminal), authenticates sudo **once, interactively** (`sudo -v` — password or fingerprint), installs a one-line sudoers drop-in (`/etc/sudoers.d/fedora-init`: `Defaults timestamp_type=global`, visudo-checked before install, guarded by content compare; the first install flips the record type, so that run authenticates a second time to write a global record), starts the `sudo -n -v` keepalive loop, installs the toolchain guarded by `rpm -q` (ansible-core, the community.general and ansible.posix collection rpms, python3-libdnf5, python3-psutil, rsync — all stock Fedora packages), maps bare arguments onto tags by substring against `roles/*` dir names, and runs `ansible-playbook site.yml --diff` with **no become password at all**: every per-task sudo authenticates via the global timestamp record, skipping PAM entirely. This design is forced, not chosen: on fingerprint-enrolled Fedora, pam_fprintd runs before pam_unix in sudo's PAM stack and — probe-verified in ansible's exact become context (pty stdin, piped stdio) — blocks silently in the fingerprint wait without ever emitting a password prompt, so become-with-password times out no matter how the password is delivered; and sudo's default tty-scoped timestamps can never match become's fresh throwaway ptys, which is why the drop-in scopes the record to the user (trade-off: while a record is live, any same-user process can sudo without re-auth — acceptable on this single-user machine, documented in README, reverted by deleting the drop-in).
- `site.yml` — one play: `hosts: localhost`, local connection, `gather_facts: false`, **`become: false` at play level**, `force_handlers: true` (a handler notification lost to a mid-play abort is never re-fired by a converged re-run). A `package_facts` pre-task (tagged `always`, `check_mode: false`) is the rpm -q guard layer. Roles run in the old NN-module order; each is tagged with its role name, underscores as hyphens.
- `roles/<name>/` — one concern per role: `tasks/main.yml`, optional `handlers/main.yml`, assets in `files/`. `roles/common/tasks/gnome_extension.yml` is a shared include (parameters: `ext_uuid`, `ext_role`, `ext_has_schemas`) used by window_snapping and no_overview.
- `library/` — two custom check-mode-aware modules for the D-Bus mutations nothing stock covers: `mutter_display_scale.py` (DisplayConfig `ApplyMonitorsConfig`) and `login_keyring.py` (gnome-keyring `ChangeWithMasterPassword`). Both must run as the desktop user on the session bus, never under become.
- `ansible.cfg` / `hosts.ini` — local inventory, `gathering = explicit`, in-repo `roles_path`/`library`.

## Conventions (enforced by review, not tooling)

- **One concern per role — and an add-on belongs to its host's role.** Something that only exists to extend another thing this repo installs goes in that thing's role, not its own: the caveman plugin lives in claude_code, podman-tui/podman-docker live in podman. The test is dependency direction: if the candidate is meaningless without the host, merge it. Do NOT merge siblings that merely share machinery (window_snapping and no_overview both use the gnome_extension include but are separate concerns) or incidental coupling (mise's shell hook rides the zsh role's zshrc asset, but a runtime manager isn't shell config). When merging costs a tag, the substring matcher usually still covers it (`./install.sh claude` reaches the caveman tasks).
- **become is opt-in per task**, only for system mutations (dnf, /etc, systemd, usermod). Anything touching the user session — dconf writes, session D-Bus, `$HOME` files, user-level installers — must NOT have become: run as root it would configure root's desktop, the classic failure the old bash modules guarded with "no sudo here" comments. Never set become at play level.
- **Zero-network converged re-run** is a hard requirement — with exactly two deliberate exceptions: the updates role (drift from upstream is undetectable offline) and podman's always-latest install (aaron's call; see both gotchas). The dnf5 module refreshes repo metadata even for no-op transactions, so every package task guards on `ansible_facts.packages` (exact rpm names — virtual provides can't false-match). Same discipline for anything else that would touch the network when nothing changed (the oh-my-zsh clone gates on its payload file; the tarball/curl installers gate on presence probes).
- **Check-mode honesty**: `./install.sh --check` on a converged machine must print every task ok/skipped with zero diffs — and on a *fresh* machine it must not crash. Read-only probes that later conditionals depend on get `changed_when: false` + `check_mode: false` (so they still run under --check). Mutating commands get an explicit `changed_when`. Registered results of tasks *skipped* in check mode lack attributes — guard downstream dereferences (`foo.path is defined`), the pattern the tempfile-using roles follow. Two fresh-`--check` gates recur: dnf5 resolves package specs even in check mode, so an install whose repo file was only *simulated* needs `not (ansible_check_mode and <repo_copy> is changed)` (brave, vscode) or `not ansible_check_mode or <release rpm> in ansible_facts.packages` (multimedia); and systemd_service fails on units whose package install was only simulated (battery).
- `gather_facts` is false: `ansible_env` does not exist. Use `{{ lookup('env', 'HOME') }}` (controller == target under the local connection).
- dconf values are **GVariant text**: strings `"'prefer-dark'"`, booleans `"true"`, doubles `"0.19"`, empty string arrays `"@as []"` (a bare `[]` is untyped and invalid), non-empty `"['a', 'b']"`. When a value comes from a YAML list var, render it explicitly (`| string`) — a bare single-variable template resolves to a native list, not text.
- Comment density is a feature: the task files carry the WHY comments the bash modules had. Keep lint clean (`.ansible-lint` skips exactly two style rules, with reasons in that file); prefer a `# noqa: rule` with justification over weakening a task.

## The GNOME extensions

`roles/window_snapping/files/rectangle@amamparo/` and `roles/no_overview/files/no-overview@amamparo/` are bundled, self-authored GNOME Shell extensions (ESM, GNOME 48–51), not vendored third-party ones — edit them directly here. The shared `gnome_extension.yml` mirrors them into `~/.local/share/gnome-shell/extensions/` with rsync; its rsync_opts are load-bearing for convergence (`--exclude=gschemas.compiled` — exists only in the installed copy; `--checksum` — tarball and git-checkout mtimes differ for identical content; `--omit-dir-times` — glib-compile-schemas rewrites schemas/ and bumps the dir mtime, which would re-report changed and re-compile forever). Schemas compile only when the sync changed (rectangle only). Enablement **merges additively** into `/org/gnome/shell/enabled-extensions`: the effective list is probed via `gsettings get` (NOT `dconf read` — vendor defaults like Fedora's background-logo live in gschema overrides, invisible in the user db) and the repo's `gnome_extensions_enabled` (site.yml) is unioned in — never overwritten, matching the old `gnome-extensions enable` append semantics; the "shell hasn't rescanned" race is gone because the write path IS the old fallback.

For rectangle: keybindings are declared in `schemas/*.gschema.xml` (the `as`-typed keys), consumed by name in `extension.js` (`KEYS`); `_unmaximize()` carries a GNOME ≤48 vs ≥49 API fallback. `metadata.json`'s `shell-version` gates which GNOME versions load an extension, and Wayland can't hot-reload GNOME Shell — extension changes take effect at logout/login.

## Gotchas encoded in the roles

- updates: sanctioned exception to the offline-converged-re-run rule, and it runs FIRST — later roles then resolve against fresh metadata. Its very first task writes `max_parallel_downloads=8` into `/etc/dnf/dnf.conf` (`community.general.ini_file`, become): the ONLY safe parallelism in this single-host play. libdnf5 downloads 3 packages at a time by default; every dnf transaction in the repo inherits the bump, so download phases overlap while the serial rpm transactions (one lock) never do — order untouched, check-mode honest, idempotent. Cross-TASK concurrency was rejected on evidence: async `poll: 0` is the only intra-host mechanism and it STARTS the real background job even under `--check` (verified: `started=True` in a dry run), which would execute installs during `./install.sh --check` — a hard violation of check-mode honesty. `forks` does nothing with one host; `pipelining` is ssh-only (local connection here). 8 is the Fedora-mirror sweet spot (past ~10, mirrors throttle). `update_cache: true` is what actually delivers the fresh-metadata promise: `state: latest` alone trusts any cache inside libdnf5's expiry window, and hours-stale metadata referencing mirror-pruned rpms 404'd podman-tui twice before this was understood. `state: latest` on `*` carries a `package-latest` noqa — latest is the concern here and in podman, nowhere else. fwupd exit codes (verified live): `get-updates` rc 0 = updates available, rc 2 = nothing to do — hence `failed_when: rc not in [0, 2]`; `update --assume-yes --no-reboot-check` applies without prompting, and staged firmware takes effect at reboot (the role says so). fwupd/flatpak tasks are guarded on the packages existing — Workstation ships both, minimal installs may not.
- steam: from RPM Fusion nonfree, so it MUST stay ordered after multimedia (which installs the release rpms); carries the same fresh-`--check` gate on `rpmfusion-nonfree-release`.
- battery: `tlp` declares `Conflicts: tuned` (`tuned-ppd` merely depends on tuned). The stock stack is removed *before* installing so its orphaned deps get swept — but the dnf5 *module*, unlike the CLI, only sweeps with `autoremove: true` (it wires `clean_requirements_on_remove` to that flag, default off); the removals here and in brave set it. Removals are guarded per exact rpm name because `tlp-pd` *provides* `power-profiles-daemon`. Unlike the dnf5 CLI, the module no-ops when removing an absent package — the guards still matter for the offline re-run. `ansible.builtin.systemd_service` fails on units whose file doesn't exist (LoadState probe, even under --check), so power-profiles-daemon.service — whose unit vanished with the package — is masked via plain `systemctl mask` guarded by `creates:` on the mask symlink; the systemd-rfkill units ship with systemd itself and use the module, but enable/start of tlp.service+tlp-pd.service is gated `not ansible_check_mode or <rpm> in ansible_facts.packages` — on a fresh --check the install was only simulated and the module would hard-fail on the missing units. TLP's docs require masking systemd-rfkill. `tlp start` is a handler (fires only on change, where the bash module ran it every time; `force_handlers` keeps a notification from being lost to a later role's failure). powertop is measurement-only — no autotune service, it would fight TLP over the same sysfs knobs.
- window_snapping / no_overview: extension install + enablement is the shared `gnome_extension.yml` (see above). GNOME's stock bindings collide with all four tiling keys (`shift-overview-up/down` defaults to Super+Alt+Up/Down; `switch-to-workspace-left/right` includes Super+Alt+Left/Right) and Mutter resolves duplicates nondeterministically — the role clears the overview pair and rebinds workspace switching to Super+Alt+PageUp/PageDown. Check both `org.gnome.shell.keybindings` and `org.gnome.desktop.wm.keybindings` schemas before adding any new binding. Super+Alt+F is GNOME's native `toggle-maximized`, bound *alongside* its stock Alt+F10 (a bare set would silently drop it); `toggle-fullscreen` is reset (dconf `state: absent`) because an earlier revision bound it to the same key. For no_overview: opening the overview at login is hardcoded in gnome-shell (upstream gnome-shell!4009 unmerged as of GNOME 50); since GNOME 50 extensions initialize after the startup animation begins, so a brief flash is unavoidable. Don't add a second hide-overview mechanism alongside it — running two is a reported crash source.
- zsh: oh-my-zsh via the git module, but gated on the payload file `oh-my-zsh.sh` (a dir without it is a broken half-install — moved aside to `~/.oh-my-zsh.broken` and recloned), NOT left to the git module's own idempotency: for an existing clone the module fetches origin every run, which would break the offline re-run and silently become an updater. The one-time `~/.zshrc.pre-fedora-init` backup (README references the name) is a cmp probe + guarded copy, not `backup: true` (which timestamps).
- git_workspace: the ssh key is presence-guarded via `creates:` — NEVER regenerate an existing key (it may be enrolled with GitHub; silent replacement breaks every remote). No passphrase, deliberately (the login-keyring LUKS trade-off). The pubkey debug is gated `not ansible_check_mode` — under fresh --check the file doesn't exist to read. Nautilus bookmarks live in `~/.config/gtk-3.0/bookmarks` even for GTK4 Nautilus ("URI [label]" lines, picked up live); the bookmark is grep-probed by URI and only *added* when absent — a lineinfile regexp rewrite would reset a user-renamed label, the Toolbox-autostart trap again. **SSH commit signing** rides here too: git's SSH backend signs commits+tags with the SAME ed25519 key (no separate GPG key), set via `community.general.git_config` `scope: global` — plus the global `user.name`/`user.email` a fresh gitconfig lacks (signing needs an author identity). All `~/.gitconfig`/`$HOME` writes, NO become (root's gitconfig otherwise — the dconf-as-root class). The `allowed_signers` file (local `git verify-commit`) is guarded on a `stat` of the pubkey (`check_mode: false`, so truthful under --check) — `lookup('file', …pub)` in its `content` would crash a fresh --check where the key was only simulated, the same trap the pubkey debug dodges. GitHub "Verified" needs the key re-added as a *Signing* key (separate slot, same key) — not automatable, so it's not attempted.
- github_cli: `gh` from stock Fedora repos (`updates`) — plain dnf5 + package-facts guard, NO vendor `.repo` and so NO fresh-`--check` repo gate (contrast vscode/tailscale). Separate role, not folded into git_workspace: gh authenticates via its own token/HTTPS flow independent of the ssh key, so the add-on merge test fails (like window_snapping vs no_overview). `gh auth login` is a browser/device flow, not automatable — surfaced as a debug nudge (gated `is changed and not ansible_check_mode`), the tailscale-`up` pattern. Substring quirk: `./install.sh gh` also matches ghostty (tag "ghostty" contains "gh"); `./install.sh github` hits only this role.
- appindicator: GNOME dropped legacy tray icons in 3.26; apps that still publish one (claude_desktop's TrayIconLinux.png, the tailscale applet) show nothing without this. Fedora ships `gnome-shell-extension-appindicator` (Workstation dep, UUID `appindicatorsupport@rgcjonas.gmail.com`, in `/usr/share/gnome-shell/extensions`) but DISABLED — so the role guards the package then enables the UUID. Deliberately does NOT use the shared `gnome_extension.yml` include: that rsyncs a bundled source tree + compiles schemas, and there's nothing to mirror for a system-installed extension — only the enablement half applies, reproduced here. Enablement is the same ADDITIVE `gsettings`-probe + `dconf` union as gnome_extension.yml (never a bare set — would drop Fedora's background-logo and the repo's rectangle/no-overview), user dconf so NO become, takes effect at next login (Wayland can't hot-reload the shell).
- multimedia: the old `dnf swap` non-idempotency becomes install-guarded-on-target + a guarded removal of the stock package. `allowerasing` alone equals `dnf swap` only for real Conflicts pairs (ffmpeg vs ffmpeg-free); the Intel driver pair *coexists* (Provides only, disjoint paths), so `libva-intel-media-driver` gets an explicit `state: absent` after the install — after, so there is never a driverless window. RPM Fusion release rpms install by URL, guarded per repo (a joint guard would re-download both when one is missing) and **need `disable_gpg_check: true`**: the dnf5 CLI leaves `localpkg_gpgcheck` off by default (why the bash flow worked) but the module forces it on, and a fresh rpmdb has no RPM Fusion key — the release rpms then install the repo keys, so subsequent rpmfusion packages ARE gpg-checked. The Fedora version comes from `lookup('pipe', 'rpm -E %fedora')` because facts are off (controller == target; change to a distribution fact if this ever targets remote hosts). intel-media-driver is hardware-specific: Intel Lunar Lake T14s Gen 6, not AMD.
- brave: default browser via `community.general.xdg_mime` (needs community.general ≥ 10.7.0; Fedora 44 ships 12.x) — as the user, never become (it writes `~/.config/mimeapps.list`). The default is set *before* the removals so http/https handlers never dangle; removals stay guarded (`item in ansible_facts.packages`) for the offline re-run. PWAs install via the Chromium `WebAppInstallForceList` policy in `/etc/brave/policies/managed/` (path verified by grepping the brave binary — Brave's equivalent of chromium's `/etc/chromium/policies`): works even for sites without a manifest (shortcut-style apps), no `create_desktop_shortcut` (that key means a ~/Desktop icon; the app-grid launcher entry is part of the install regardless), and force-listed apps can't be uninstalled from the UI — removal happens by dropping the entry from `files/policies.json` and re-running. Chromium hot-reloads platform policy files, so a running Brave applies changes in seconds; when Brave isn't running the role nudges it awake (`--no-startup-window --no-first-run`, backgrounded, killed afterwards — only the instance the role spawned, never the user's) and waits for the `brave-*-Default.desktop` entries to appear. That count-based wait is a deliberate heuristic: app ids are manifest-derived and unknowable offline, so hand-installed PWAs can satisfy the count (under-trigger, never over-trigger).
- gnome_prefs: everything as the user, no become (root's dconf otherwise). Display scale is `library/mutter_display_scale.py`: monitors.xml is owned by mutter's DisplayConfig D-Bus API (method 2 = persist + apply live); it applies only when the built-in panel (`is-builtin` monitor prop — a logical-monitor count would misfire docked lid-closed) is the sole active display, because positions are scale-dependent logical pixels and mutter *rejects* overlapping re-scaled layouts ("Logical monitors overlap") rather than repositioning, which would fail the play. Docked layouts are separate monitors.xml configs anyway.
- ghostty: from the scottames COPR (the repo ghostty's official install docs name for Fedora — a community build, gpg-signed with its COPR key; `files/ghostty.repo` is the COPR-served file verbatim, `skip_if_unavailable` included). App-search-for-"terminal" works because the packaged desktop entry ships `Keywords=terminal;tty;pty;` — verified by extracting the rpm; no desktop-file override exists, so don't add one. Ptyxis is removed *after* ghostty installs (never leave the machine terminal-less — the brave ordering rule for terminals); removal is what delivers "not accessible, not in search". The Moonfly theme moved here from the retired gnome_prefs Ptyxis tasks — bundled with ghostty (verified in the rpm), set in the repo-owned `files/config`.
- login_keyring: the keyring's key derives from the login password; fingerprint can never unlock it (the sensor yields no key material). The role blanks the keyring password via `library/login_keyring.py` (gnome-keyring's `InternalUnsupportedGuiltRiddenInterface.ChangeWithMasterPassword` — internal but stable, gnome-initial-setup relies on it), which needs the *current* password: a **gated `pause` task** prompts only when the keyring is present-and-encrypted and never under --check (`vars_prompt` was rejected — it fires unconditionally at play start). State detection is file magic, no D-Bus: plaintext `[keyring]` = blank, binary `GnomeKeyring` = encrypted. The password rides a `no_log` module arg over the local connection's stdin (never argv/env). Wrong password fails the task with the Seahorse guidance, nothing modified. Cleanup of the earlier revision's greeter drop-ins (`/etc/dconf/db/gdm.d/…`) is gated on the keyring being confirmed blank — a failed blanking never restores fingerprint login prematurely — followed *immediately* by a `dconf update` gated on `is changed` (the old `removed=1` flag; deliberately not a handler — a deferred flush lost to a later role's failure would leave the compiled gdm db stale with nothing on disk for a re-run to notice). Rejected alternatives: greeter password-only (user wants fingerprint), `authselect disable-feature with-fingerprint` (gates only sudo/polkit), true fingerprint unlock (impossible; TPM-backed work hasn't shipped in Fedora 44).
- vscode: `roles/vscode/files/vscode.repo` is Microsoft's documented content verbatim — `autorefresh`/`type` are zypper-isms dnf5 silently ignores; keep the file byte-diffable against the docs, and don't install it via `dnf config-manager addrepo` (rejects `autorefresh`, errors on existing files). No `rpm --import`: under Fedora 44's rpm 6, imported keys are named by *full fingerprint*, so the classic `gpg-pubkey-<short-keyid>` guard never matches — the repo file's `gpgkey` + `gpgcheck=1` lets dnf import and verify on install. Microsoft's documented `dnf check-update` step stays dropped (exit 100 when updates exist). The 2025 Microsoft key rotation doesn't affect this repo — yumrepos/vscode still signs with the original `microsoft.asc`.
- jetbrains_toolbox: Toolbox 3.x is a JVM app bundle — no rpm, no FUSE (that was the ≤2.6.2 AppImage era), no sudo anywhere: the installer derives the target home from `getpwuid()`, ignoring `$HOME`, so become would install into root's home. `--install` is undocumented (the docs claim silent Linux install is impossible) but verified: full headless self-install. Only the bundle copy into `Toolbox/bin` is synchronous — daemon, desktop entries, and autostart land via a background child seconds later, so the post-install assert checks only the binary, while the *re-run* guard probes the binary AND the applications `.desktop` entry (a reboot in that window leaves a half-install a binary-only guard would wedge forever; the applications entry, not autostart — users may turn launch-at-login off). Tarball URL from the data.services.jetbrains.com 302 redirect; sha256 sidecar verified with the hash passed literally to get_url (its checksum-URL parser and the sidecar's `<hash> *<filename>` format don't mix). Toolbox self-updates in place: presence is terminal, no version probing.
- claude_code: the native installer has no already-installed check (re-downloads the full ~260 MB binary every unguarded run) — the role guards on `~/.local/bin/claude` (a stable symlink into `~/.local/share/claude/versions/`; the binary self-updates, so presence is terminal) plus a `command -v claude` probe (an npm/dnf install counts; docs warn against stacking copies — and don't probe `~/.claude`, that's config/state shared with the IDE extensions). The installer is downloaded to a temp file then run — never piped into bash (a mid-stream death would execute a truncated script). `get_url` needs `force: true` there because tempfile pre-creates the (empty) dest and get_url's default would skip the download. tempfile has no check-mode support: downstream tasks guard on `.path is defined`. **ultracode default** rides here too (last tasks in the role): the settings schema makes `ultracode` SESSION-SCOPED — the in-session toggle never persists it — so the only way to default it is a persistent settings source, `~/.claude/settings.json`. The role MERGES the single `ultracode: true` key (slurp -> `from_json` -> `combine` -> `to_nice_json` copy), never owning the file: Claude Code rewrites settings.json at runtime (model/theme/effort are interactively settable and persisted), so a whole-file copy would fight the app on every interactive change — the same rewrite-at-runtime reason claude_desktop leaves `claude_desktop_config.json` unmanaged. Idempotency gates on the LOGICAL value (`not (claude_settings.ultracode | default(false) | bool)`), NOT file bytes: the canonical `to_nice_json` render won't match Claude Code's own on-disk formatting, so a bytes/checksum comparison would report changed and rewrite every run, flip-flopping against the app forever; value-gating lands the one-time reformat once and skips thereafter (ultracode is a recognized schema key, so the app preserves it across its rewrites). Runs after the caveman install so the slurp reads that write and re-merges its `enabledPlugins`/marketplace keys forward. slurp is `failed_when: false` (file absent on a fresh machine -> parse defaults to `{}`); `$HOME` file, NO become. Needs workflows enabled (default on-plan) + an xhigh-capable `model` (opus qualifies).
- podman: chosen over Docker (Fedora-native, daemonless, rootless — no root-equivalent docker group; the repo's brave-style "one vendor repo fewer" logic too). The package task is deliberately unguarded `state: latest` + `update_cache: true` (aaron wants podman always current; the second network exception). `podman.socket` is enabled at `scope: user` — session bus, NO become, same rule as dconf — and gated `not ansible_check_mode or 'podman' in ansible_facts.packages` (systemd_service fails on missing units even under --check). `/etc/containers/nodocker` uses `state: touch` + `modification_time/access_time: preserve`, the idempotent-touch pattern — a bare touch would report changed every run. Fedora already provisions rootless subuid/subgid ranges at user creation; don't manage them.
- tailscale: `files/tailscale.repo` is the vendor's file verbatim (both `gpgcheck` and `repo_gpgcheck` are Tailscale's own settings). Login is not automatable — `tailscale up` is a browser flow — so the role probes `tailscale status --json` for `"NeedsLogin"` (BackendState) and prints the one manual step; the probe chain is gated on the binary existing so fresh `--check` stays clean. dnf + systemd tasks carry the standard fresh-`--check` gates.
- reaper: no rpm/repo exists — official tarball's `install-reaper.sh --install /opt --integrate-sys-desktop --usr-local-bin-symlink --quiet` (flags verified against the script), gated on `/opt/REAPER/reaper` (presence is terminal: no self-update on Linux, the updates role can't see it — the jetbrains rule). "Latest" resolves at install time from `reaper.fm/latestversion/` (plain text, version is the FIRST line — changelog follows) and the tarball URL is derived (`7.77` -> `files/7.x/reaper777_linux_x86_64.tar.xz`); x86_64 hardcoded (single-machine repo, the intel-media-driver precedent). ALL resource-path work (`~/.config/REAPER`) is user-session — NO become (root's REAPER otherwise). Theme install = copy the `.ReaperThemeZip` (REAPER loads it zipped, never extract) + `lastthemefn5` in the ini seed; `libSwell-user.colortheme` in the resource-path root is the Linux dark-dialogs file. Icon packs merge with rsync `--checksum --omit-dir-times` but deliberately NO `--delete` (unlike gnome_extension): `Data/toolbar_icons`/`track_icons` are user-extended pools, a mirror would delete hand-imported icons. `files/toolbar_icons` is a pre-merged union of the theme pack + Essential Icons "Style Reapertips Theme" + 3 unique files from that pack's `Update/` dir (verified: zero filename overlap between packs; Update/'s other 123 files are byte-identical to the main pack). SWS installs by scraping the homepage's featured `-Linux-x86_64.tar.xz` href (regex verified against live page; no /latest endpoint exists) and overlaying the tarball onto the resource path (its layout IS a resource-path overlay: UserPlugins/Scripts/Data); ReaPack uses GitHub's stable `/releases/latest/download/` URL. Both gate on the `.so` existing. `reaper.ini` is SEEDED ONLY WHEN ABSENT (template + stat guard): REAPER rewrites it on every exit — the claude_desktop_config.json rewrite-at-runtime trap — so converged management would flip-flop forever; after first run the file belongs to REAPER. Every seeded key was verified against mespotine's config-var docs (github.com/mespotine/ultraschall-and-reaper-docs); PDF suggestions whose keys could NOT be verified (mouse modifiers, folder collapse, track spacer, MIDI editor bits — 7.x features postdating the docs, or opaque bitfields like `deftrackrecflags` whose other bits carry unknown defaults) are deliberately a one-time debug checklist (nudge gated `reaper_seeded is changed and not ansible_check_mode`), never guessed. Bitfield traps encoded in the template comments: `splitautoxfade` &8 is INVERTED (1 = fades off), `recupdatems` is 1000/Hz not Hz, `loadlastproj` new-project is 19. **Reversed horizontal scroll** (touchpad-natural panning) is a `reaper-kb.ini` keymap binding, NOT a mouse modifier (arrange wheel isn't an mm_ctx context — binary has TCP/MCP wheel contexts only): `KEY 255 216 <cmd> <section>` = bare HorizWheel (216 verified against mespotine's filetype docs AND real-world keymaps; 88 is a legacy alias — don't use it), cmd 977 = Main reversed scroll, 40660 = MIDI Editor (section 32060). reaper-kb.ini is app-rewritten like reaper.ini BUT preserves adopted lines, so it gets the ADDITIVE-probe treatment (grep for any `KEY 255 216 * <section>` binding, add only when none — an in-app rebinding must win, the bookmark rule), not seed-if-absent. Theme/icons are CC BY-NC-SA 4.0 (redistribution in this repo is licensed; attribution file installed to Data/). Fonts: Roboto via dnf (stock), Fira Sans bundled ttfs (`dnf5 search fira` is empty on Fedora 44) into `~/.local/share/fonts` + fc-cache handler.
- mise: replaces pyenv/nvm/jenv (installs and pins runtimes; reads `.nvmrc`/`.python-version`). `files/mise.repo` is the vendor's file verbatim — note the baseurl is `mise.en.dev` while docs live at `mise.jdx.dev` (hosting move, both endpoints verified live; don't "fix" the URL). The shell hook lives in `roles/zsh/files/zshrc`, guarded on `command -v mise` — it must stay in the zsh role's asset: a lineinfile from the mise role would diverge `~/.zshrc` from the asset and the zsh role's cmp-copy would clobber it on the next run. Runtimes are deliberately not provisioned (per-project `mise use` decisions).
- claude_code / caveman plugin (same role — pure add-on, merged): the plugin installs via `claude plugin marketplace add` + `claude plugin install` as the user — state in `~/.claude/plugins/` (known_marketplaces.json, marketplaces/, cache/, installed_plugins.json). The marketplace source is the **https URL**, not the `owner/repo` shorthand: the shorthand clones over `git@github.com:`, which fails on a fresh machine whose key isn't enrolled with GitHub yet. The `claude plugin list` probe has `failed_when: false` and its rc gates the plugin tasks — under fresh `--check` the CLI install above was only simulated, so the binary doesn't exist and a hard probe would crash the dry run. Guards probe *presence*, not enabled-state — installing enables the plugin, and re-enabling every run would fight a deliberate `claude plugin disable`. `claude` runs with `~/.local/bin` prepended to PATH (the play's PATH may lack it before the zshrc takes effect at next login).
- claude_desktop: separate role from claude_code, NOT an add-on — the GUI chat app is meaningful without the CLI (the reason it's here is local stdio MCP, which a PWA/Electron-wrapper of claude.ai can't do), and neither depends on the other, so the merge test fails (contrast the caveman plugin, which IS meaningless without the CLI). Anthropic ships no Fedora build — the official Linux beta is Debian/Ubuntu-only ("Fedora and RHEL: only Debian-based distributions are supported today") — so this installs `claude-desktop-unofficial` from aaddrick/claude-desktop-debian's signed DNF repo, which since v3.0.0 repackages Anthropic's *official* Linux `.deb` (thin repack, low breakage risk vs the pre-v3 Windows-app-extraction era). Pure vendor-repo role (vscode/tailscale pattern): repo file + dnf5, package-facts guard + fresh-`--check` gate. Updates ride the `updates` role's `dnf upgrade` — the app doesn't self-update on Linux (Anthropic's docs; dnf is the intended channel), so no self-updater to fight. Two decisions encoded: (1) `files/claude-desktop-unofficial.repo` is the vendor file verbatim EXCEPT an added `skip_if_unavailable=1` — deliberate deviation from the file-verbatim convention because it's a single-maintainer self-hosted domain and the updates role runs `dnf upgrade` on `*`, so a dead host would hard-fail every system-update run (the ghostty COPR file is the precedent for a repo file carrying this flag). (2) The install guards on the exact rpm name `claude-desktop-unofficial`, never the virtual `claude-desktop` it Provides — virtual-provides matching is forbidden by convention, and a future official Anthropic rpm named `claude-desktop` would false-match and silently skip. gpgcheck=1 + repo_gpgcheck=1 with the gpgkey URL in the file, so dnf imports/verifies the maintainer's key on install — no rpm --import (the vscode rpm-6-fingerprint lesson). MCP config lives at `~/.config/Claude/claude_desktop_config.json` — the role deliberately does NOT manage it (per-user server choices, and the app rewrites the file on exit). Substring-matcher note: `./install.sh claude` now matches BOTH claude_code and claude_desktop (both dir names contain "claude") — running it reaches both roles; use `claude-desktop` / `claude-code` to target one. Trust: unofficial-squared (unofficial packaging of an app Anthropic doesn't ship for Fedora), maintainer's GPG key not Anthropic's — accepted deliberately; the rpm's `Provides: claude-desktop` eases a future migration to an official Fedora rpm. **Window↔launcher binding fix**: the role also drops `files/claude-desktop.desktop` into `~/.local/share/applications/` under the SAME basename as the vendor entry (`claude-desktop-unofficial.desktop`) so it fully shadows `/usr/share`'s copy (highest-precedence XDG data dir; .desktop entries replace rather than merge across dirs, and dnf upgrades only rewrite `/usr/share`, so the override survives). Reason: GNOME binds a live window to its launcher by matching the window's WM_CLASS (X11) / app_id (Wayland) against `StartupWMClass` then a basename fallback, but the official Electron build advertises `claude-desktop` — Chromium's `GetXdgAppId()` reads the asar's `desktopName: "claude-desktop.desktop"` and IGNORES the launcher's `--class=Claude` (aaddrick #588) — which matches NEITHER the vendor entry's `StartupWMClass=Claude` nor its `-unofficial` basename, so the running window shows a nameless/icon-less "claude-desktop" even though app-search looks right (search reads Name=/Icon= off disk, no live window). The override's only delta from the vendor file is `StartupWMClass=claude-desktop`. It's a `$HOME`/user-session file — NEVER become (root's copy binds nothing); `update-desktop-database` runs only on change (copy is the idempotency gate). Takes effect on the next quit+relaunch of Claude (already-open windows keep their stale binding; no logout needed — that's only for shell extensions).
