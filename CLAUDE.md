# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Keep documentation in sync

When a change alters behavior, commands, role conventions, keybindings, or gotchas, update the documentation in the same change — this file, README.md, and any task-file header comments that describe the affected behavior. README.md is the user-facing doc (what the roles do and how to run them); CLAUDE.md is the contributor-facing doc (architecture, conventions, gotchas). A PR that changes what a role does but not the docs is incomplete.

## What this is

An Ansible playbook that configures a fresh Fedora Workstation (GNOME) install, targeting Fedora 44 / GNOME 50. The only entry point is:

```sh
./install.sh                # bootstrap the toolchain + run every role
./install.sh battery zsh    # run roles whose tag contains a substring
./install.sh --check        # dry run (any dash-arg passes through to ansible-playbook)
curl -fsSL https://raw.githubusercontent.com/amamparo/fedora-init/main/install.sh | bash   # no checkout
```

**Running the playbook mutates the host** (installs/removes dnf packages, masks systemd units, changes the login shell). Don't execute it to "test" a change unless the user asks — even `--check` prompts for sudo. The safe checks are:

```sh
ansible-playbook site.yml --syntax-check
ansible-lint --offline        # policy in .ansible-lint; passes the production profile
bash -n install.sh
python3 -m py_compile library/*.py
```

## Architecture

- `install.sh` — orchestrator and bootstrapper. If it can't find `site.yml` next to itself (piped from curl, or a stray lone copy), it downloads the GitHub tarball to a temp dir and re-runs from that copy. Otherwise it: reattaches stdin to `/dev/tty` (under curl|bash stdin is the exhausted pipe; the sudo prompt and the keyring pause both need the terminal), authenticates sudo **once, interactively** (`sudo -v` — password or fingerprint), installs a one-line sudoers drop-in (`/etc/sudoers.d/fedora-init`: `Defaults timestamp_type=global`, visudo-checked before install, guarded by content compare; the first install flips the record type, so that run authenticates a second time to write a global record), starts the `sudo -n -v` keepalive loop, installs the toolchain guarded by `rpm -q` (ansible-core, the community.general and ansible.posix collection rpms, python3-libdnf5, python3-psutil, rsync — all stock Fedora packages), maps bare arguments onto tags by substring against `roles/*` dir names, and runs `ansible-playbook site.yml --diff` with **no become password at all**: every per-task sudo authenticates via the global timestamp record, skipping PAM entirely. This design is forced, not chosen: on fingerprint-enrolled Fedora, pam_fprintd runs before pam_unix in sudo's PAM stack and — probe-verified in ansible's exact become context (pty stdin, piped stdio) — blocks silently in the fingerprint wait without ever emitting a password prompt, so become-with-password times out no matter how the password is delivered; and sudo's default tty-scoped timestamps can never match become's fresh throwaway ptys, which is why the drop-in scopes the record to the user (trade-off: while a record is live, any same-user process can sudo without re-auth — acceptable on this single-user machine, documented in README, reverted by deleting the drop-in).
- `site.yml` — one play: `hosts: localhost`, local connection, `gather_facts: false`, **`become: false` at play level**, `force_handlers: true` (a handler notification lost to a mid-play abort is never re-fired by a converged re-run). A `package_facts` pre-task (tagged `always`, `check_mode: false`) is the rpm -q guard layer. Roles run in the old NN-module order; each is tagged with its role name, underscores as hyphens.
- `roles/<name>/` — one concern per role: `tasks/main.yml`, optional `handlers/main.yml`, assets in `files/`. `roles/common/tasks/gnome_extension.yml` is a shared include (parameters: `ext_uuid`, `ext_role`, `ext_has_schemas`) used by window_snapping and no_overview.
- `library/` — two custom check-mode-aware modules for the D-Bus mutations nothing stock covers: `mutter_display_scale.py` (DisplayConfig `ApplyMonitorsConfig`) and `login_keyring.py` (gnome-keyring `ChangeWithMasterPassword`). Both must run as the desktop user on the session bus, never under become.
- `ansible.cfg` / `hosts.ini` — local inventory, `gathering = explicit`, in-repo `roles_path`/`library`.

## Conventions (enforced by review, not tooling)

- **One concern per role — and an add-on belongs to its host's role.** Something that only exists to extend another thing this repo installs goes in that thing's role, not its own: the caveman plugin lives in claude_code, podman-tui/podman-docker live in podman. The test is dependency direction: if the candidate is meaningless without the host, merge it. Do NOT merge siblings that merely share machinery (window_snapping and no_overview both use the gnome_extension include but are separate concerns) or incidental coupling (mise's shell hook rides the zsh role's zshrc asset, but a runtime manager isn't shell config). When merging costs a tag, the substring matcher usually still covers it (`./install.sh claude` reaches the caveman tasks).
- **become is opt-in per task**, only for system mutations (dnf, /etc, systemd, usermod). Anything touching the user session — dconf writes, session D-Bus, `$HOME` files, user-level installers — must NOT have become: run as root it would configure root's desktop, the classic failure the old bash modules guarded with "no sudo here" comments. Never set become at play level.
- **Zero-network converged re-run** is a hard requirement. The dnf5 module refreshes repo metadata even for no-op transactions, so every package task guards on `ansible_facts.packages` (exact rpm names — virtual provides can't false-match). Same discipline for anything else that would touch the network when nothing changed (the oh-my-zsh clone gates on its payload file; the tarball/curl installers gate on presence probes).
- **Check-mode honesty**: `./install.sh --check` on a converged machine must print every task ok/skipped with zero diffs — and on a *fresh* machine it must not crash. Read-only probes that later conditionals depend on get `changed_when: false` + `check_mode: false` (so they still run under --check). Mutating commands get an explicit `changed_when`. Registered results of tasks *skipped* in check mode lack attributes — guard downstream dereferences (`foo.path is defined`), the pattern the tempfile-using roles follow. Two fresh-`--check` gates recur: dnf5 resolves package specs even in check mode, so an install whose repo file was only *simulated* needs `not (ansible_check_mode and <repo_copy> is changed)` (brave, vscode) or `not ansible_check_mode or <release rpm> in ansible_facts.packages` (multimedia); and systemd_service fails on units whose package install was only simulated (battery).
- `gather_facts` is false: `ansible_env` does not exist. Use `{{ lookup('env', 'HOME') }}` (controller == target under the local connection).
- dconf values are **GVariant text**: strings `"'prefer-dark'"`, booleans `"true"`, doubles `"0.19"`, empty string arrays `"@as []"` (a bare `[]` is untyped and invalid), non-empty `"['a', 'b']"`. When a value comes from a YAML list var, render it explicitly (`| string`) — a bare single-variable template resolves to a native list, not text.
- Comment density is a feature: the task files carry the WHY comments the bash modules had. Keep lint clean (`.ansible-lint` skips exactly two style rules, with reasons in that file); prefer a `# noqa: rule` with justification over weakening a task.

## The GNOME extensions

`roles/window_snapping/files/rectangle@amamparo/` and `roles/no_overview/files/no-overview@amamparo/` are bundled, self-authored GNOME Shell extensions (ESM, GNOME 48–51), not vendored third-party ones — edit them directly here. The shared `gnome_extension.yml` mirrors them into `~/.local/share/gnome-shell/extensions/` with rsync; its rsync_opts are load-bearing for convergence (`--exclude=gschemas.compiled` — exists only in the installed copy; `--checksum` — tarball and git-checkout mtimes differ for identical content; `--omit-dir-times` — glib-compile-schemas rewrites schemas/ and bumps the dir mtime, which would re-report changed and re-compile forever). Schemas compile only when the sync changed (rectangle only). Enablement **merges additively** into `/org/gnome/shell/enabled-extensions`: the effective list is probed via `gsettings get` (NOT `dconf read` — vendor defaults like Fedora's background-logo live in gschema overrides, invisible in the user db) and the repo's `gnome_extensions_enabled` (site.yml) is unioned in — never overwritten, matching the old `gnome-extensions enable` append semantics; the "shell hasn't rescanned" race is gone because the write path IS the old fallback.

For rectangle: keybindings are declared in `schemas/*.gschema.xml` (the `as`-typed keys), consumed by name in `extension.js` (`KEYS`); `_unmaximize()` carries a GNOME ≤48 vs ≥49 API fallback. `metadata.json`'s `shell-version` gates which GNOME versions load an extension, and Wayland can't hot-reload GNOME Shell — extension changes take effect at logout/login.

## Gotchas encoded in the roles

- battery: `tlp` declares `Conflicts: tuned` (`tuned-ppd` merely depends on tuned). The stock stack is removed *before* installing so its orphaned deps get swept — but the dnf5 *module*, unlike the CLI, only sweeps with `autoremove: true` (it wires `clean_requirements_on_remove` to that flag, default off); the removals here and in brave set it. Removals are guarded per exact rpm name because `tlp-pd` *provides* `power-profiles-daemon`. Unlike the dnf5 CLI, the module no-ops when removing an absent package — the guards still matter for the offline re-run. `ansible.builtin.systemd_service` fails on units whose file doesn't exist (LoadState probe, even under --check), so power-profiles-daemon.service — whose unit vanished with the package — is masked via plain `systemctl mask` guarded by `creates:` on the mask symlink; the systemd-rfkill units ship with systemd itself and use the module, but enable/start of tlp.service+tlp-pd.service is gated `not ansible_check_mode or <rpm> in ansible_facts.packages` — on a fresh --check the install was only simulated and the module would hard-fail on the missing units. TLP's docs require masking systemd-rfkill. `tlp start` is a handler (fires only on change, where the bash module ran it every time; `force_handlers` keeps a notification from being lost to a later role's failure). powertop is measurement-only — no autotune service, it would fight TLP over the same sysfs knobs.
- window_snapping / no_overview: extension install + enablement is the shared `gnome_extension.yml` (see above). GNOME's stock bindings collide with all four tiling keys (`shift-overview-up/down` defaults to Super+Alt+Up/Down; `switch-to-workspace-left/right` includes Super+Alt+Left/Right) and Mutter resolves duplicates nondeterministically — the role clears the overview pair and rebinds workspace switching to Super+Alt+PageUp/PageDown. Check both `org.gnome.shell.keybindings` and `org.gnome.desktop.wm.keybindings` schemas before adding any new binding. Super+Alt+F is GNOME's native `toggle-maximized`, bound *alongside* its stock Alt+F10 (a bare set would silently drop it); `toggle-fullscreen` is reset (dconf `state: absent`) because an earlier revision bound it to the same key. For no_overview: opening the overview at login is hardcoded in gnome-shell (upstream gnome-shell!4009 unmerged as of GNOME 50); since GNOME 50 extensions initialize after the startup animation begins, so a brief flash is unavoidable. Don't add a second hide-overview mechanism alongside it — running two is a reported crash source.
- zsh: oh-my-zsh via the git module, but gated on the payload file `oh-my-zsh.sh` (a dir without it is a broken half-install — moved aside to `~/.oh-my-zsh.broken` and recloned), NOT left to the git module's own idempotency: for an existing clone the module fetches origin every run, which would break the offline re-run and silently become an updater. The one-time `~/.zshrc.pre-fedora-init` backup (README references the name) is a cmp probe + guarded copy, not `backup: true` (which timestamps).
- git_workspace: the ssh key is presence-guarded via `creates:` — NEVER regenerate an existing key (it may be enrolled with GitHub; silent replacement breaks every remote). No passphrase, deliberately (the login-keyring LUKS trade-off). The pubkey debug is gated `not ansible_check_mode` — under fresh --check the file doesn't exist to read. Nautilus bookmarks live in `~/.config/gtk-3.0/bookmarks` even for GTK4 Nautilus ("URI [label]" lines, picked up live); the bookmark is grep-probed by URI and only *added* when absent — a lineinfile regexp rewrite would reset a user-renamed label, the Toolbox-autostart trap again.
- multimedia: the old `dnf swap` non-idempotency becomes install-guarded-on-target + a guarded removal of the stock package. `allowerasing` alone equals `dnf swap` only for real Conflicts pairs (ffmpeg vs ffmpeg-free); the Intel driver pair *coexists* (Provides only, disjoint paths), so `libva-intel-media-driver` gets an explicit `state: absent` after the install — after, so there is never a driverless window. RPM Fusion release rpms install by URL, guarded per repo (a joint guard would re-download both when one is missing) and **need `disable_gpg_check: true`**: the dnf5 CLI leaves `localpkg_gpgcheck` off by default (why the bash flow worked) but the module forces it on, and a fresh rpmdb has no RPM Fusion key — the release rpms then install the repo keys, so subsequent rpmfusion packages ARE gpg-checked. The Fedora version comes from `lookup('pipe', 'rpm -E %fedora')` because facts are off (controller == target; change to a distribution fact if this ever targets remote hosts). intel-media-driver is hardware-specific: Intel Lunar Lake T14s Gen 6, not AMD.
- brave: default browser via `community.general.xdg_mime` (needs community.general ≥ 10.7.0; Fedora 44 ships 12.x) — as the user, never become (it writes `~/.config/mimeapps.list`). The default is set *before* the removals so http/https handlers never dangle; removals stay guarded (`item in ansible_facts.packages`) for the offline re-run.
- gnome_prefs: everything as the user, no become (root's dconf otherwise). The Ptyxis profile uuid doesn't exist until Ptyxis first launches — the palette write is probe-gated and may need a re-run. Display scale is `library/mutter_display_scale.py`: monitors.xml is owned by mutter's DisplayConfig D-Bus API (method 2 = persist + apply live); it applies only when the built-in panel (`is-builtin` monitor prop — a logical-monitor count would misfire docked lid-closed) is the sole active display, because positions are scale-dependent logical pixels and mutter *rejects* overlapping re-scaled layouts ("Logical monitors overlap") rather than repositioning, which would fail the play. Docked layouts are separate monitors.xml configs anyway.
- login_keyring: the keyring's key derives from the login password; fingerprint can never unlock it (the sensor yields no key material). The role blanks the keyring password via `library/login_keyring.py` (gnome-keyring's `InternalUnsupportedGuiltRiddenInterface.ChangeWithMasterPassword` — internal but stable, gnome-initial-setup relies on it), which needs the *current* password: a **gated `pause` task** prompts only when the keyring is present-and-encrypted and never under --check (`vars_prompt` was rejected — it fires unconditionally at play start). State detection is file magic, no D-Bus: plaintext `[keyring]` = blank, binary `GnomeKeyring` = encrypted. The password rides a `no_log` module arg over the local connection's stdin (never argv/env). Wrong password fails the task with the Seahorse guidance, nothing modified. Cleanup of the earlier revision's greeter drop-ins (`/etc/dconf/db/gdm.d/…`) is gated on the keyring being confirmed blank — a failed blanking never restores fingerprint login prematurely — followed *immediately* by a `dconf update` gated on `is changed` (the old `removed=1` flag; deliberately not a handler — a deferred flush lost to a later role's failure would leave the compiled gdm db stale with nothing on disk for a re-run to notice). Rejected alternatives: greeter password-only (user wants fingerprint), `authselect disable-feature with-fingerprint` (gates only sudo/polkit), true fingerprint unlock (impossible; TPM-backed work hasn't shipped in Fedora 44).
- vscode: `roles/vscode/files/vscode.repo` is Microsoft's documented content verbatim — `autorefresh`/`type` are zypper-isms dnf5 silently ignores; keep the file byte-diffable against the docs, and don't install it via `dnf config-manager addrepo` (rejects `autorefresh`, errors on existing files). No `rpm --import`: under Fedora 44's rpm 6, imported keys are named by *full fingerprint*, so the classic `gpg-pubkey-<short-keyid>` guard never matches — the repo file's `gpgkey` + `gpgcheck=1` lets dnf import and verify on install. Microsoft's documented `dnf check-update` step stays dropped (exit 100 when updates exist). The 2025 Microsoft key rotation doesn't affect this repo — yumrepos/vscode still signs with the original `microsoft.asc`.
- jetbrains_toolbox: Toolbox 3.x is a JVM app bundle — no rpm, no FUSE (that was the ≤2.6.2 AppImage era), no sudo anywhere: the installer derives the target home from `getpwuid()`, ignoring `$HOME`, so become would install into root's home. `--install` is undocumented (the docs claim silent Linux install is impossible) but verified: full headless self-install. Only the bundle copy into `Toolbox/bin` is synchronous — daemon, desktop entries, and autostart land via a background child seconds later, so the post-install assert checks only the binary, while the *re-run* guard probes the binary AND the applications `.desktop` entry (a reboot in that window leaves a half-install a binary-only guard would wedge forever; the applications entry, not autostart — users may turn launch-at-login off). Tarball URL from the data.services.jetbrains.com 302 redirect; sha256 sidecar verified with the hash passed literally to get_url (its checksum-URL parser and the sidecar's `<hash> *<filename>` format don't mix). Toolbox self-updates in place: presence is terminal, no version probing.
- claude_code: the native installer has no already-installed check (re-downloads the full ~260 MB binary every unguarded run) — the role guards on `~/.local/bin/claude` (a stable symlink into `~/.local/share/claude/versions/`; the binary self-updates, so presence is terminal) plus a `command -v claude` probe (an npm/dnf install counts; docs warn against stacking copies — and don't probe `~/.claude`, that's config/state shared with the IDE extensions). The installer is downloaded to a temp file then run — never piped into bash (a mid-stream death would execute a truncated script). `get_url` needs `force: true` there because tempfile pre-creates the (empty) dest and get_url's default would skip the download. tempfile has no check-mode support: downstream tasks guard on `.path is defined`.
- podman: chosen over Docker (Fedora-native, daemonless, rootless — no root-equivalent docker group; the repo's brave-style "one vendor repo fewer" logic too). `podman.socket` is enabled at `scope: user` — session bus, NO become, same rule as dconf — and gated `not ansible_check_mode or 'podman' in ansible_facts.packages` (systemd_service fails on missing units even under --check). `/etc/containers/nodocker` uses `state: touch` + `modification_time/access_time: preserve`, the idempotent-touch pattern — a bare touch would report changed every run. Fedora already provisions rootless subuid/subgid ranges at user creation; don't manage them.
- tailscale: `files/tailscale.repo` is the vendor's file verbatim (both `gpgcheck` and `repo_gpgcheck` are Tailscale's own settings). Login is not automatable — `tailscale up` is a browser flow — so the role probes `tailscale status --json` for `"NeedsLogin"` (BackendState) and prints the one manual step; the probe chain is gated on the binary existing so fresh `--check` stays clean. dnf + systemd tasks carry the standard fresh-`--check` gates.
- mise: replaces pyenv/nvm/jenv (installs and pins runtimes; reads `.nvmrc`/`.python-version`). `files/mise.repo` is the vendor's file verbatim — note the baseurl is `mise.en.dev` while docs live at `mise.jdx.dev` (hosting move, both endpoints verified live; don't "fix" the URL). The shell hook lives in `roles/zsh/files/zshrc`, guarded on `command -v mise` — it must stay in the zsh role's asset: a lineinfile from the mise role would diverge `~/.zshrc` from the asset and the zsh role's cmp-copy would clobber it on the next run. Runtimes are deliberately not provisioned (per-project `mise use` decisions).
- claude_code / caveman plugin (same role — pure add-on, merged): the plugin installs via `claude plugin marketplace add` + `claude plugin install` as the user — state in `~/.claude/plugins/` (known_marketplaces.json, marketplaces/, cache/, installed_plugins.json). The marketplace source is the **https URL**, not the `owner/repo` shorthand: the shorthand clones over `git@github.com:`, which fails on a fresh machine whose key isn't enrolled with GitHub yet. The `claude plugin list` probe has `failed_when: false` and its rc gates the plugin tasks — under fresh `--check` the CLI install above was only simulated, so the binary doesn't exist and a hard probe would crash the dry run. Guards probe *presence*, not enabled-state — installing enables the plugin, and re-enabling every run would fight a deliberate `claude plugin disable`. `claude` runs with `~/.local/bin` prepended to PATH (the play's PATH may lack it before the zshrc takes effect at next login).
